# ПОЯСНЮВАЛЬНА ЗАПИСКА

## АНОТАЦІЯ

Мета даного курсового проекту – проектування та реалізація інформаційної системи для організації та автоматизації процесів взаємодії між замовниками та фрілансерами на спеціалізованій веб-платформі "FreelanceHub". Користувачами даної системи є замовники, що публікують проекти та шукають виконавців, фрілансери, що пропонують свої послуги у сферах веб-розробки, дизайну, копірайтингу та маркетингу, а також адміністратори, що здійснюють модерацію платформи.

Реалізація виконана з використанням мови JavaScript, а саме фреймворку веб-розробки Next.js, СУБД PostgreSQL та бібліотеки node-postgres (pg), що надає можливість прямого виконання SQL-запитів до бази даних без використання ORM. Авторизацію та автентифікацію користувачів забезпечено за допомогою бібліотеки NextAuth.js з використанням стратегії Credentials та хешування паролів через bcrypt. Стилізацію інтерфейсу виконано з використанням CSS-фреймворку Tailwind CSS. Архітектура системи побудована за принципами трирівневої клієнт-серверної архітектури.

У розробленій інформаційній системі передбачено повний життєвий цикл проекту від його публікації замовником до завершення угоди з виконавцем, управління фінансовими гарантіями через систему ескроу-рахунків, рейтингову систему оцінювання фрілансерів та механізм подання та розгляду скарг. Відмінною особливістю даної системи є забезпечення безпеки на рівні бази даних через механізм Row-Level Security (RLS), чотири рівні ролей доступу (гість, замовник, виконавець, адміністратор) та автоматичне оновлення рейтингів виконавців за допомогою тригерів бази даних.

Результатом курсового проектування є інформаційна система фрілансерської платформи, що має адаптивний, простий та інтуїтивний інтерфейс користувача українською мовою.

## ВСТУП

Стрімкий розвиток цифрової економіки та поширення віддаленої роботи призвели до значного зростання ринку фрілансу. Все більше компаній та приватних осіб звертаються до незалежних спеціалістів для виконання проектних робіт у сферах веб-розробки, дизайну, копірайтингу та маркетингу. Водночас зростає і кількість фахівців, які обирають фріланс як основну або додаткову форму зайнятості.

Наразі будь-який замовник, що потребує виконання певних робіт, стикається з необхідністю пошуку кваліфікованого виконавця, узгодження умов співпраці, контролю виконання завдань та забезпечення фінансових гарантій для обох сторін. Відсутність єдиної системи для організації цих процесів призводить до ризиків невиконання зобов'язань, втрати коштів та конфліктних ситуацій між учасниками. Тому проектування, створення та впровадження інформаційної системи, що дозволяє не тільки зберігати інформацію про проекти та учасників, а й забезпечувати безпечну взаємодію між ними, є актуальною задачею.

На ринку існують відомі фрілансерські платформи, такі як Upwork, Freelancer та Fiverr, проте жодна з них не орієнтована на українськомовну аудиторію та не враховує специфіку локального ринку послуг. Крім того, більшість існуючих рішень є закритими комерційними продуктами, що не дозволяє адаптувати їх під конкретні потреби.

Предметна область, що розглядається в цій роботі – фрілансерська платформа, яка забезпечує взаємодію між замовниками проектів та незалежними виконавцями у сферах веб-розробки, дизайну, копірайтингу та маркетингу.

Мета курсового проекту – проектування та реалізація інформаційної системи для організації та автоматизації процесів взаємодії між замовниками та фрілансерами. Створювана система надає можливості:

- замовникам публікувати проекти із зазначенням спеціалізації, бюджету та термінів виконання, переглядати каталог фрілансерів, укладати угоди з обраними виконавцями та залишати відгуки після завершення співпраці;
- фрілансерам переглядати доступні проекти, отримувати пропозиції від замовників, виконувати завдання в рамках укладених угод та відстежувати власний рейтинг;

## 1 ПОСТАНОВКА ЗАДАЧІ

Предметна область, що розглядається в цій роботі – фрілансерська платформа, яка забезпечує взаємодію між замовниками проектів та незалежними виконавцями у сферах веб-розробки, дизайну, копірайтингу та маркетингу.

При аналізі даної предметної області сформульовано основні завдання, які має вирішувати дана ІС:

1) публікація проектів замовниками із зазначенням спеціалізації, опису, бюджету та терміну виконання;
2) ведення каталогу фрілансерів з можливістю фільтрації за спеціалізацією та містом;
3) організація повного життєвого циклу угоди від її створення до завершення з відповідною зміною статусів;
4) забезпечення фінансових гарантій через систему ескроу-рахунків, що автоматично заморожують кошти при укладанні угоди та вивільняють їх після успішного завершення;
5) ведення рейтингової системи оцінювання фрілансерів з автоматичним перерахунком середнього рейтингу;
6) забезпечення механізму подання та розгляду скарг між учасниками платформи;
7) адміністрування платформи з можливістю перегляду статистики, управління користувачами та розгляду скарг.

Як результат аналізу та детального розгляду ІС фрілансерської платформи виявлено 4 основні групи користувачів, а саме:

1) гості (неавторизовані користувачі) – мають право переглядати головну сторінку платформи, каталог опублікованих проектів із можливістю фільтрації за спеціалізацією та бюджетом, а також каталог зареєстрованих фрілансерів із можливістю фільтрації за спеціалізацією та містом;
2) замовники (клієнти) – мають право створювати, редагувати та видаляти власні проекти, переглядати профілі фрілансерів та укладати з ними угоди, відстежувати стан власних угод та змінювати їх статуси, переглядати ескроу-рахунки за своїми угодами, залишати відгуки та оцінки фрілансерам після завершення співпраці, подавати скарги на виконавців, а також редагувати власний профіль;
3) фрілансери (виконавці) – мають право переглядати доступні проекти, відстежувати стан угод, в яких вони є виконавцями, та змінювати їх статуси, переглядати ескроу-рахунки за своїми угодами, подавати скарги на замовників, а також редагувати власний профіль із зазначенням спеціалізації, резюме та портфоліо;
4) адміністратори – мають право переглядати статистику платформи через агреговані представлення бази даних, управляти обліковими записами користувачів (перегляд та видалення), переглядати рейтинг найкращих фрілансерів, розглядати та змінювати статуси скарг від усіх учасників платформи, а також переглядати всі угоди та ескроу-рахунки в системі.

## 2 ПРОЕКТУВАННЯ ІНФОРМАЦІЙНОЇ СИСТЕМИ

При аналізі структури необхідного додатку обрано трирівневу архітектуру.

Трирівнева архітектура – це архітектурна модель, що передбачає в її складі три взаємопов'язані компоненти: клієнт, що зазвичай виступає веб-браузером, сервер додатку та сервер баз даних (рис. 2.1).

> **Рисунок 2.1 – Трирівнева архітектура**
>
> *Промпт для Gemini: "Створи просту схему трирівневої клієнт-серверної архітектури українською мовою. Три блоки вертикально: верхній — «Клієнт (веб-браузер)», середній — «Сервер додатку (Next.js)», нижній — «Сервер баз даних (PostgreSQL)». Між блоками двонаправлені стрілки з підписами «HTTP-запити/відповіді» та «SQL-запити/результати». Стиль: мінімалістична діаграма для курсового проекту, чорно-білий."*

У такій архітектурі кожний компонент створює свій логічний рівень: клієнт відповідає за рівень представлення, сервер додатку за рівень бізнес-логіки та сервер баз даних за рівень доступу до даних [1] (рис. 2.2).

> **Рисунок 2.2 – Логічні рівні додатку**
>
> *Промпт для Gemini: "Створи діаграму логічних рівнів веб-додатку українською мовою. Три горизонтальні шари: верхній — «Рівень представлення (Presentation Layer)» з підписом «React-компоненти, Tailwind CSS», середній — «Рівень бізнес-логіки (Business Logic Layer)» з підписом «Next.js API Routes, NextAuth.js», нижній — «Рівень доступу до даних (Data Access Layer)» з підписом «PostgreSQL, node-postgres (pg)». Стрілки між шарами. Стиль: мінімалістична діаграма для курсового проекту, чорно-білий."*

При такій архітектурі окремі компоненти ізолюють логіку один від одного, що підвищує гнучкість та полегшує розширення і розробку кожного компоненту окремо, оскільки такий підхід дозволяє працювати над кожним рівнем різним командам розробників.

Така архітектура накладає певні обов'язки та правила на свої компоненти:

1) клієнт – це компонент верхнього рівня, що надає користувачеві можливість взаємодії з даними без прямих зв'язків з сервером бази даних. У даній системі клієнтом виступає веб-браузер, який отримує від сервера HTML-сторінки, сформовані за допомогою React-компонентів та стилізовані з використанням Tailwind CSS;
2) веб-сервер – це компонент середнього рівня, що надає користувачеві основну бізнес-логіку системи. У даній системі сервер реалізовано на основі фреймворку Next.js, який обробляє HTTP-запити через API Routes, виконує авторизацію за допомогою NextAuth.js та забезпечує розмежування доступу на основі ролей;
3) сервер баз даних – це компонент нижнього рівня, який надає можливість збереження та обробки даних, що реалізується засобами СУБД PostgreSQL. На цьому рівні забезпечено додатковий захист даних через механізм Row-Level Security (RLS), тригери для автоматичного оновлення рейтингів та представлення (VIEW) для агрегації статистичних даних.

Виходячи з архітектури, необхідно обрати шаблон проектування додатку. Фреймворк Next.js за своєю природою реалізує архітектурний підхід, наближений до шаблону MVC (Model-View-Controller) (рис. 2.3), який остаточно описано у 1988 році. Популярність шаблону підтверджується наявністю багатьох функціональних фреймворків та бібліотек, серед яких відомі AngularJS, ASP.NET MVC, PHP Laravel, Ruby on Rails та Next.js.

Конкретніше про кожну із компонент шаблону у контексті розробленої системи:

1) «модель» (Model) визначає, які дані має містити додаток. У розробленій системі роль моделі виконує база даних PostgreSQL з 10 таблицями, 3 представленнями, 5 функціями та тригерами. Взаємодія з моделлю здійснюється через бібліотеку node-postgres (pg) шляхом прямого виконання параметризованих SQL-запитів;
2) «представлення» (View) визначає, як мають відображатися дані програми. У розробленій системі представлення реалізовано за допомогою React-компонентів, які формують 14 сторінок веб-інтерфейсу з адаптивним дизайном на основі Tailwind CSS;
3) «контролер» (Controller) містить логіку, яка оновлює модель та представлення у відповідь на вхідні дані від користувачів програми. У розробленій системі роль контролерів виконують API Routes фреймворку Next.js, що обробляють HTTP-запити (GET, POST, PUT, DELETE), виконують валідацію даних, перевіряють права доступу та взаємодіють з базою даних.

> **Рисунок 2.3 – Модель MVC у контексті FreelanceHub**
>
> *Промпт для Gemini: "Створи діаграму шаблону MVC для веб-додатку FreelanceHub українською мовою. Три блоки: «Model» (підпис: PostgreSQL, 10 таблиць, pg), «View» (підпис: React-компоненти, 14 сторінок, Tailwind CSS), «Controller» (підпис: Next.js API Routes, NextAuth.js). Стрілки: Користувач → Controller (HTTP-запит), Controller → Model (SQL-запит), Model → Controller (дані), Controller → View (відповідь), View → Користувач (HTML-сторінка). ## 3 ІНФОРМАЦІЙНЕ МОДЕЛЮВАННЯ ПРЕДМЕТНОЇ ОБЛАСТІ

Сутності та їх атрибути з описом обмежень, що потрібні для розв'язання поставлених задач, наведено в додатку B.

В базі даних інформаційної системи «FreelanceHub» існують 3 види зв'язку між сутностями, а саме:

1) один-до-одного;
2) один-до-багатьох;
3) багато-до-багатьох.

Розглянемо кожний зі зв'язків по черзі:

1) зв'язок один-до-одного формалізується додаванням первинного ключа однієї сутності у якості унікального зовнішнього ключа до іншої сутності. Такий вид зв'язку наявний між таблицями:

a) Users → Client — кожний обліковий запис користувача з роллю «client» має рівно один профіль замовника. Зв'язок реалізовано через поле user_id у таблиці Client з обмеженням UNIQUE та зовнішнім ключем на Users(user_id) з каскадним видаленням;
b) Users → Contractor — кожний обліковий запис користувача з роллю «contractor» має рівно один профіль виконавця. Зв'язок реалізовано аналогічно через поле user_id у таблиці Contractor з обмеженням UNIQUE та каскадним видаленням;

2) зв'язок один-до-багатьох формалізується додаванням первинного ключа сутності, що відповідає одиничному зв'язку, у якості зовнішнього ключа до сутності зі сторони множинного зв'язку. Такий вид зв'язку наявний між таблицями:

a) Client → Project — один замовник може створити багато проектів, але кожний проект належить одному замовнику;
b) Client → Deal — один замовник може мати багато угод;
c) Contractor → Deal — один виконавець може мати багато угод;
d) Project → Deal — за одним проектом може бути укладено багато угод;
e) Deal → Safe — за однією угодою може бути створено багато ескроу-рахунків;
f) Deal → ContractorTask — за однією угодою може бути створено багато завдань;
g) Contractor → ContractorTask — один виконавець може мати багато завдань;
h) Client → Ratings (як рецензент) — один замовник може залишити багато відгуків;
i) Contractor → Ratings (як оцінюваний) — один виконавець може отримати багато відгуків;
j) Project → Ratings — за одним проектом може бути залишено багато відгуків;
k) Client → ComplaintClient (як відправник) — один замовник може подати багато скарг;
l) Contractor → ComplaintClient (як отримувач) — на одного виконавця може надійти багато скарг;
m) Contractor → ComplaintContractor (як відправник) — один виконавець може подати багато скарг;
n) Client → ComplaintContractor (як отримувач) — на одного замовника може надійти багато скарг;

3) зв'язок багато-до-багатьох формалізується створенням додаткової сутності та доданням первинних ключів обох сутностей до неї. Подібний зв'язок наявний між наступними сутностями:

a) Client ↔ Contractor, для формалізації яких було створено сутність Deal, яка виступає логічною сутністю «Угода між замовником та виконавцем». Один замовник може укладати угоди з багатьма виконавцями, і один виконавець може мати угоди з багатьма замовниками;
b) Client ↔ Contractor (через відгуки), для формалізації яких було створено сутність Ratings, яка виступає логічною сутністю «Відгук замовника про виконавця». Один замовник може оцінити багатьох виконавців, і один виконавець може бути оцінений багатьма замовниками;
c) Client ↔ Contractor (через скарги від замовників), для формалізації яких було створено сутність ComplaintClient, яка виступає логічною сутністю «Скарга замовника на виконавця»;
d) Contractor ↔ Client (через скарги від виконавців), для формалізації яких було створено сутність ComplaintContractor, яка виступає логічною сутністю «Скарга виконавця на замовника»;
e) Contractor ↔ Deal, для формалізації яких було створено сутність ContractorTask з унікальним обмеженням UNIQUE(contractor_id, deal_id), яка виступає логічною сутністю «Завдання виконавця за угодою». Один виконавець може мати завдання за багатьма угодами, і одна угода може мати завдання для багатьох виконавців.

Наприклад, розглянемо зв'язок між сутностями Client та Contractor через сутність Deal. Кожний замовник може укладати угоди з декількома виконавцями на виконання різних проектів, при цьому кожний виконавець також може мати угоди з декількома замовниками. Таким чином, маємо зв'язок багато-до-багатьох, який формалізовано через сутність Deal, що містить зовнішні ключі на обидві сутності (client_id та contractor_id), а також зовнішній ключ на Project (project_id).

Наступний приклад, розглянемо зв'язок між сутностями Users та Client. Зрозуміло, що кожний обліковий запис користувача може мати лише один профіль замовника, а кожний профіль замовника належить лише одному обліковому запису. Отже маємо класичний зв'язок один-до-одного, реалізований через унікальний зовнішній ключ user_id.

Усі вказані таблиці знаходяться у формі Бойса – Кодда.

## 4 ВИБІР ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ ДЛЯ СТВОРЕННЯ ІНФОРМАЦІЙНОЇ СИСТЕМИ

ІС фрілансерської платформи «FreelanceHub» розроблена у форматі веб-додатку, тобто користувач може взаємодіяти з системою просто зайшовши з браузера. Система реалізована із залученням наступних технологій:

1) для взаємодії з базою даних обрано СУБД PostgreSQL 18;
2) основна мова програмування додатку – JavaScript, яка використовується як на стороні клієнта (React-компоненти), так і на стороні сервера (API Routes);
3) Front-End – бібліотека React 19 із серверними та клієнтськими компонентами у межах фреймворку Next.js та CSS-фреймворк Tailwind CSS v4 для стилізації інтерфейсу;
4) Back-End – повностековий фреймворк Next.js 16 з використанням App Router та API Routes для обробки HTTP-запитів;
5) бібліотека node-postgres (pg) для прямого виконання параметризованих SQL-запитів до бази даних без використання ORM;
6) бібліотека NextAuth.js 4 для авторизації та автентифікації з використанням стратегії Credentials;
7) бібліотека bcryptjs для хешування паролів користувачів.

СУБД PostgreSQL обрано через декілька основних причин:

1) дана СУБД є вільно поширюваною та безкоштовною за базовою ліцензією, що не потребує додаткових фінансових витрат;
2) наявна велика кількість документації у відкритому доступі, що спрощує процес розробки та налагодження;
3) об'єктно-реляційна модель представлення даних дозволяє використовувати розширені типи даних, зокрема ENUM для статусів угод, скарг та ескроу-рахунків, а також JSONB для зберігання структурованих вхідних та вихідних даних завдань виконавців;
4) має гнучку та зручну структуру управління та організації рольової системи, що дозволяє створювати ролі на рівні бази даних (freelance_admin, freelance_client, freelance_contractor, freelance_guest) з налаштуванням привілеїв через GRANT/REVOKE;
5) підтримує механізм Row-Level Security (RLS), що дозволяє обмежити доступ користувачів до конкретних рядків таблиць на основі їхньої ролі;
6) підтримує тригери та збережені функції мовою PL/pgSQL, що використано для автоматичного оновлення рейтингів виконавців та перевірки бізнес-логіки.

Фреймворк Next.js обрано як основу серверної та клієнтської частини додатку. Next.js є повностековим React-фреймворком, що забезпечує серверний рендеринг (SSR), маршрутизацію на основі файлової системи (App Router) та вбудований механізм API Routes для створення серверних ендпоінтів. Використання єдиного фреймворку для Front-End та Back-End значно спрощує архітектуру додатку, оскільки не потребує окремого серверного фреймворку. App Router забезпечує автоматичну маршрутизацію: кожний файл `page.js` у директорії `app/` автоматично стає окремою сторінкою, а файли `route.js` — API-ендпоінтами, що обробляють HTTP-запити.

CSS-фреймворк Tailwind CSS обрано для стилізації інтерфейсу користувача. На відміну від традиційних CSS-фреймворків, таких як Bootstrap, Tailwind CSS використовує підхід «utility-first», що надає великий набір готових CSS-класів для безпосереднього застосування в HTML-розмітці. Це дозволяє швидко створювати адаптивний дизайн без написання власних CSS-файлів та значно прискорює розробку інтерфейсу.

Для взаємодії з базою даних обрано бібліотеку node-postgres (pg), яка забезпечує прямий доступ до PostgreSQL через виконання параметризованих SQL-запитів. На відміну від ORM-фреймворків (таких як Prisma або Sequelize), які генерують SQL-запити автоматично, використання node-postgres дозволяє повністю контролювати структуру запитів, використовувати специфічні можливості PostgreSQL (VIEW, функції, тригери, RLS) та оптимізувати продуктивність запитів. Бібліотека підтримує пул з'єднань (Pool), що дозволяє ефективно розподіляти підключення до бази даних між одночасними запитами.

Бібліотеку NextAuth.js обрано для реалізації авторизації та автентифікації користувачів. NextAuth.js є спеціалізованим рішенням для Next.js, що підтримує різні стратегії автентифікації. У розробленій системі використано стратегію Credentials, що дозволяє авторизувати користувачів за логіном та паролем. Хешування паролів здійснюється за допомогою бібліотеки bcryptjs, що реалізує алгоритм bcrypt з сіллю та налаштовуваною кількістю раундів хешування.

Для розробки додатку використано середовище Visual Studio Code — безкоштовний редактор коду від компанії Microsoft з підтримкою розширень для JavaScript, React та роботи з базами даних.

## 5 ПРОГРАМНА МОДЕЛЬ ЗАСТОСУНКА

Модель взаємодії рівня бізнес-логіки представлена набором API Routes — серверних обробників HTTP-запитів, що виконують роль контролерів у шаблоні MVC. Кожний API Route є окремим файлом `route.js`, який експортує асинхронні функції, що відповідають HTTP-методам GET, POST, PUT та DELETE, та дозволяють клієнтській частині «спілкуватись» з сервером через інтерфейсні елементи (кнопки, форми, посилання). Маршрутизація запитів здійснюється автоматично на основі файлової структури директорії `app/api/`.

Структура додатку складається з 13 API-ендпоінтів (route-файлів), згрупованих за предметними областями. Кожний ендпоінт обробляє запити, що стосуються конкретної сутності або групи сутностей бази даних. Розмежування доступу до методів здійснюється на рівні кожного обробника шляхом перевірки ролі поточного користувача з об'єкту сесії NextAuth.

Пул з'єднань автоматично розподіляє підключення до бази даних між одночасними запитами.

На відміну від ORM-фреймворків, де кожна таблиця бази даних транслюється у окремий клас-модель, у розробленій системі дані з бази отримуються безпосередньо у вигляді JavaScript-об'єктів через виконання SQL-запитів. Наприклад, для отримання списку проектів API Route `/api/projects` виконує SQL-запит з JOIN-операцією між таблицями Project та Client, а результат повертається клієнту у форматі JSON без проміжного перетворення у класи.

Такий підхід обрано з наступних причин:

1) повний контроль над структурою SQL-запитів, що дозволяє використовувати специфічні можливості PostgreSQL (ENUM-типи, JSONB, VIEW, функції);
2) можливість оптимізації складних запитів з JOIN-операціями, агрегатними функціями та фільтрацією;
3) відсутність надлишкового шару абстракції, що спрощує налагодження та розуміння логіки доступу до даних;
4) пряме використання механізмів безпеки PostgreSQL — Row-Level Security (RLS) та рольової системи доступу.

Порівняння структури таблиці бази даних та відповідного SQL-запиту в API Route наведено на рис. 5.2.

> **Рисунок 5.2 — Порівняння таблиці Project та SQL-запиту у API Route**
>
> *Промпт для Gemini: "Створи порівняльну діаграму для курсового проекту українською мовою. Зліва блок «Таблиця Project (PostgreSQL)» з полями: project_id SERIAL PK, client_id INT FK, specialization project_specialization, description TEXT, budget DECIMAL(10,2), deadline DATE, created_at TIMESTAMP, updated_at TIMESTAMP. Рівень представлення (View) реалізовано у вигляді набору React-компонентів, що формують веб-сторінки додатку. Кожна сторінка є окремим файлом `page.js` у відповідній директорії маршруту. В залежності від того, під якою роллю авторизовано користувача, він отримує доступ до специфічних сторінок та функцій інтерфейсу.

Розподіл доступу до веб-сторінок за ролями користувачів:

1) **Гість** (неавторизований) — головна сторінка, каталог проектів, каталог фрілансерів, авторизація, реєстрація;
2) **Замовник** (client) — усі сторінки гостя, а також: створення проектів, редагування/видалення власних проектів, управління угодами, ескроу-рахунки, рейтинги та відгуки, скарги, профіль;
3) **Виконавець** (contractor) — усі сторінки гостя, а також: управління угодами, ескроу-рахунки, скарги, профіль з портфоліо та резюме;

## 6 СТВОРЕННЯ БАЗИ ДАНИХ

Кожна база даних складається з таких основних компонентів як таблиці, послідовності, домени, типи даних, представлення, функції та тригери.

У розробленій системі використано користувацькі типи даних (ENUM), що дозволяють обмежити множину допустимих значень для конкретних полів таблиць. Розглянемо створення типів даних для статусів угод та ролей користувачів:

```sql
CREATE TYPE deal_status AS ENUM (
    'Pending', 'In Progress', 'Completed', 'Cancelled'
);

CREATE TYPE user_role AS ENUM (
    'client', 'contractor', 'admin'
);
```

*Лістинг 6.1 — Створення користувацьких типів ENUM*

Для створення типу використовується команда CREATE TYPE, після якої вказується ім'я типу та ключове слово ENUM з переліком допустимих значень. У першому типі це статуси угоди: «очікує», «в процесі», «завершено», «скасовано». У другому — ролі користувачів системи.

Також створено домен user_login, що представляє обмеження на строкову змінну логіна користувача:

```sql
CREATE DOMAIN user_login AS VARCHAR(20)
CHECK (VALUE ~ '^[a-zA-Z0-9_.]+$');
```

*Лістинг 6.2 — Створення домену user_login*

Домен user_login обмежує поле логіна до 20 символів та дозволяє використовувати лише латинські літери, цифри, крапку та символ підкреслення. Перевірка здійснюється за допомогою регулярного виразу у CHECK-обмеженні.

Для створення таблиці Users, що зберігає облікові дані користувачів, потрібно виконати наступний SQL-запит:

```sql
CREATE TABLE Users (
    user_id SERIAL PRIMARY KEY,
    login user_login UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role user_role NOT NULL DEFAULT 'client',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

*Лістинг 6.3 — Створення таблиці Users*

Для створення таблиці використовується команда CREATE TABLE, після якої вказується назва таблиці. Для кожного поля вказується тип даних, що представляє інформацію, яку він зберігає. Поле user_id є первинним ключем (PRIMARY KEY) та має тип SERIAL, що насправді не є типом даних, а представляє собою механізм використання такого компоненту баз даних як SEQUENCE, або послідовність, що збільшується на одиницю кожний раз, коли видається нове значення за замовчуванням. Поле login використовує створений раніше домен user_login, що автоматично застосовує перевірку регулярним виразом. Обмеження UNIQUE гарантує унікальність значення в межах таблиці. Пункт DEFAULT у стовпці role означає, що якщо при створенні нового рядку дане поле буде порожнім, то за замовчуванням воно буде встановлено значенням 'client'. Обмеження NOT NULL означає, що поле не може приймати порожні значення.

Для прикладу створимо таблицю сутності Deal, що посилається на декілька інших таблиць:

```sql
CREATE TABLE Deal (
    deal_id SERIAL PRIMARY KEY,
    project_id INT REFERENCES Project(project_id) ON DELETE SET NULL,
    client_id INT REFERENCES Client(client_id) ON DELETE SET NULL,
    contractor_id INT REFERENCES Contractor(contractor_id) ON DELETE SET NULL,
    status deal_status NOT NULL DEFAULT 'Pending',
    amount DECIMAL(10,2) CHECK (amount > 0),
    creation_date DATE NOT NULL DEFAULT CURRENT_DATE,
    completion_date DATE,
    CHECK (completion_date IS NULL OR creation_date <= completion_date),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

*Лістинг 6.4 — Створення таблиці Deal*

Поля project_id, client_id та contractor_id посилаються на відповідні первинні ключі таблиць Project, Client та Contractor, виступаючи зовнішніми ключами з метою забезпечення зв'язку та цілісності. Після вказання стовпця йде умова ON DELETE SET NULL, що означає, що при видаленні батьківського запису значення зовнішнього ключа буде встановлено у NULL замість каскадного видалення. Поле status використовує раніше створений тип deal_status. Табличне обмеження CHECK гарантує, що дата завершення не може бути раніше дати створення угоди.

Для створення представлення використовується команда CREATE VIEW, після якої вказується його назва. Представлення — це віртуальна поіменована похідна таблиця, тобто її не існує у фізичному вигляді, а замість читання таблиці у базі виконується SQL-запит, який представляє це представлення.

```sql
CREATE VIEW top_contractors AS
SELECT
    c.contractor_id,
    c.first_name || ' ' || c.last_name AS full_name,
    c.specialization,
    c.rating,
    c.city,
    COUNT(d.deal_id) AS total_deals,
    COUNT(CASE WHEN d.status = 'Completed' THEN 1 END) AS completed_deals,
    ROUND(AVG(r.rating), 2) AS avg_review_rating
FROM Contractor c
LEFT JOIN Deal d ON c.contractor_id = d.contractor_id
LEFT JOIN Ratings r ON c.contractor_id = r.reviewed_id
GROUP BY c.contractor_id, c.first_name, c.last_name,
         c.specialization, c.rating, c.city
ORDER BY c.rating DESC;
```

*Лістинг 6.5 — Створення представлення top_contractors*

Дане представлення надає можливість переглянути рейтинг найкращих виконавців платформи — повне ім'я, спеціалізацію, рейтинг, місто, загальну кількість угод, кількість завершених угод та середню оцінку від замовників. Представлення використовує LEFT JOIN для з'єднання таблиць Contractor, Deal та Ratings, агрегатні функції COUNT та AVG, а також умовний підрахунок через CASE для визначення кількості завершених угод.

Тексти команд SQL-запитів, які використовуються для створення всіх типів, домену та таблиць БД наведені в додатку C.

Деякі допоміжні тригери та функції виконують необхідні завдання автоматизації процесів та слідкують за правилами цілісності ІС.

Серед них, наприклад, автоматичне оновлення рейтингу виконавця при створенні або зміні відгуку (тригер trg_update_contractor_rating, що викликає функцію update_contractor_rating), оновлення часової мітки профілю замовника після залишення відгуку, а також функції для ранжування виконавців за кількістю завершених проектів (get_ranked_contractors) та розрахунку середньої вартості години роботи над проектом (calculate_avg_hourly_cost).

## 8 ПРОГРАМНА РЕАЛІЗАЦІЯ ІНТЕРФЕЙСУ

Додаток побудовано за шаблоном MVC, тому взаємодія відбувається наступним чином:

1) користувачу надається графічний інтерфейс за допомогою React-компонентів (представлень). За допомогою сторінки авторизації він може ввести логін та пароль облікового запису та зайти з правами конкретної ролі. В залежності від ролі, яку має користувач, він може взаємодіяти з різноманітними API Routes (контролерами);
2) реакції на взаємодію кожного користувача з інтерфейсом викликають HTTP-методи GET, POST, PUT та DELETE через функцію `fetch()`, що звертаються до серверних API Routes;
3) маршрутизація здійснюється автоматично на основі файлової структури у форматі `/api/{сутність}/[id]`. Параметр id є опціональним, оскільки не кожний метод потребує ідентифікації сутності для виведення контенту;
4) потім сервер отримує дані із бази даних через модуль `lib/db.js`, виконуючи параметризовані SQL-запити, та повертає їх у форматі JSON;
5) наступним кроком React-компонент клієнтської частини отримує JSON-дані та формує вміст сторінки залежно від переданих даних, а далі браузер відображає сгенеровану сторінку користувачу. Цикл взаємодії на цьому завершується.

Звернення до бази даних виконується через модуль `lib/db.js`, що створює пул з'єднань (Pool) та експортує функцію `query(text, params)`. На відміну від ORM-фреймворків, де кожна таблиця транслюється у окремий клас-модель, у розробленій системі дані з бази отримуються безпосередньо у вигляді JavaScript-об'єктів. Вміст модуля `lib/db.js` наведений у лістингу 8.1.

```javascript
import { Pool } from 'pg'

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
})

// проста функція для запитів до бд
export async function query(text, params) {
  const res = await pool.query(text, params)
  return res
}

export default pool
```

*Лістинг 8.1 — Модуль підключення до бази даних lib/db.js*

Кожний API Route є окремим файлом, що експортує асинхронні функції GET, POST, PUT та DELETE. Ці функції виконують роль action-методів контролерів: перевіряють сесію користувача, виконують SQL-запити до бази даних та повертають результат у форматі JSON. Приклад API Route для роботи з проектами наведений у лістингу 8.2.

```javascript
import { query } from '@/lib/db'
import { getServerSession } from 'next-auth'
import { authOptions } from '@/lib/auth'
import { NextResponse } from 'next/server'

// отримати список проектів
export async function GET(req) {
  try {
    const { searchParams } = new URL(req.url)
    const specialization = searchParams.get('specialization')

    let sql = `
      SELECT p.*, c.first_name || ' ' || c.last_name AS client_name,
             c.city AS client_city
      FROM Project p
      JOIN Client c ON p.client_id = c.client_id
      WHERE 1=1
    `
    const params = []
    let i = 1

    if (specialization) {
      sql += ` AND p.specialization::text = $${i++}`
      params.push(specialization)
    }
    sql += ' ORDER BY p.created_at DESC'

    const result = await query(sql, params)
    return NextResponse.json(result.rows)
  } catch (error) {
    console.log('Projects GET error:', error)
    return NextResponse.json(
      { error: 'Помилка отримання проектів' }, { status: 500 }
    )
  }
}

// створити проект (тільки клієнт)
export async function POST(req) {
  try {
    const session = await getServerSession(authOptions)
    if (!session || session.user.role !== 'client') {
      return NextResponse.json(
        { error: 'Доступ заборонено' }, { status: 403 }
      )
    }

    const { description, specialization, budget, deadline } = await req.json()

    const result = await query(
      `INSERT INTO Project
        (client_id, specialization, description, budget, deadline)
       VALUES ($1, $2::project_specialization, $3, $4, $5) RETURNING *`,
      [session.user.profileId, specialization, description, budget, deadline]
    )

    return NextResponse.json(result.rows[0], { status: 201 })
  } catch (error) {
    console.log('Projects POST error:', error)
    return NextResponse.json(
      { error: 'Помилка створення проекту' }, { status: 500 }
    )
  }
}
```

*Лістинг 8.2 — API Route для отримання та створення проектів (/api/projects/route.js)*

Метод GET виконує виведення списку проектів з можливістю фільтрації за спеціалізацією. SQL-запит динамічно формується залежно від наявності параметрів фільтрації у URL. Метод POST виконується після відправки форми створення проекту. Спочатку перевіряється сесія користувача та його роль — тільки замовник (client) може створювати проекти. Якщо перевірка проходить, виконується SQL-запит INSERT з параметризованими значеннями ($1, $2, ...), що захищає від SQL-ін'єкцій. У разі виникнення помилок на стороні серверу баз даних виключення перехоплюється блоком try/catch та повертається відповідь з кодом помилки 500.

Алгоритм операцій оновлення та видалення працює загалом аналогічно. У цьому випадку додатково потрібно отримати ідентифікатор сутності з URL-параметра та перевірити права доступу. Кожний наступний API Route для інших сутностей (Deal, Contractor, Ratings, Complaints, Safe) побудований аналогічним чином.

Операція виводу вмісту на клієнтській стороні виконується за допомогою React-компонентів. Кожна сторінка є окремим файлом `page.js` з директивою `'use client'`, що означає виконання компонента в браузері користувача. Компонент використовує хуки React: `useState` для управління станом даних та `useEffect` для завантаження даних з API при першому рендерингу. Приклад React-компоненту сторінки каталогу проектів наведено у лістингу 8.3.

*Лістинг 8.3 — React-компонент сторінки каталогу проектів (app/projects/page.js)*

Компонент `ProjectsPage` реалізує повний цикл клієнт-серверної взаємодії: при завантаженні сторінки функція `loadProjects()` виконує GET-запит до API Route `/api/projects`, отримує JSON-масив проектів та зберігає його у стані компоненту. Кнопки фільтрації за спеціалізацією оновлюють параметр `filter`, що автоматично перезавантажує список проектів. Кнопка «Створити проект» відображається тільки для користувачів з роллю `client`, що реалізується перевіркою `session?.user.role === 'client'`. CSS-стилізація виконується за допомогою utility-класів Tailwind CSS.

Логіка автентифікації та авторизації реалізована у окремому модулі `lib/auth.js` та обробляється бібліотекою NextAuth.js. Для автентифікації використовується стратегія Credentials з JWT-токенами. При спробі авторизації виконується SQL-запит до таблиці Users для пошуку користувача за логіном, потім перевіряється хеш пароля за допомогою `bcrypt.compare()`. У разі успішної перевірки створюється JWT-токен з інформацією про роль, логін та ідентифікатор профілю, який зберігається у cookie браузера. Вміст модулю авторизації наведено у лістингу 8.4.

```javascript
import CredentialsProvider from 'next-auth/providers/credentials'
import bcrypt from 'bcryptjs'
import { query } from './db'

export const authOptions = {
  providers: [
    CredentialsProvider({
      name: 'Credentials',
      credentials: {
        login: { label: "Логін", type: "text" },
        password: { label: "Пароль", type: "password" }
      },
      async authorize(credentials) {
        if (!credentials?.login || !credentials?.password) {
          return null
        }

        try {
          // шукаємо користувача в бд
          const result = await query(
            'SELECT * FROM Users WHERE login = $1',
            [credentials.login]
          )
          const user = result.rows[0]
          if (!user) return null

          // перевіряємо пароль
          const isValid = await bcrypt.compare(
            credentials.password, user.password_hash
          )
          if (!isValid) return null

          // дістаємо дані профілю залежно від ролі
          let profile = null
          if (user.role === 'client') {
            const res = await query(
              'SELECT * FROM Client WHERE user_id = $1',
              [user.user_id]
            )
            profile = res.rows[0]
          } else if (user.role === 'contractor') {
            const res = await query(
              'SELECT * FROM Contractor WHERE user_id = $1',
              [user.user_id]
            )
            profile = res.rows[0]
          }

          return {
            id: user.user_id,
            login: user.login,
            email: user.email,
            role: user.role,
            profileId: profile
              ? (profile.client_id || profile.contractor_id)
              : null,
            firstName: profile?.first_name || 'Admin',
            lastName: profile?.last_name || '',
          }
        } catch (error) {
          console.log('Auth error:', error)
          return null
        }
      }
    })
  ],
  callbacks: {
    async jwt({ token, user }) {
      if (user) {
        token.role = user.role
        token.login = user.login
        token.profileId = user.profileId
        token.firstName = user.firstName
        token.lastName = user.lastName
      }
      return token
    },
    async session({ session, token }) {
      session.user.id = token.sub
      session.user.role = token.role
      session.user.login = token.login
      session.user.profileId = token.profileId
      session.user.firstName = token.firstName
      session.user.lastName = token.lastName
      return session
    }
  },
  pages: { signIn: '/login' },
  session: { strategy: 'jwt' },
  secret: process.env.NEXTAUTH_SECRET,
}
```

*Лістинг 8.4 — Модуль авторизації lib/auth.js*

Функція `authorize()` виконує повний цикл автентифікації: пошук користувача за логіном у таблиці Users, перевірку хешу пароля через bcrypt та завантаження профільних даних з таблиці Client або Contractor залежно від ролі. Колбеки `jwt` та `session` забезпечують передачу інформації про роль та ідентифікатор профілю у JWT-токен та об'єкт сесії відповідно. Не маючи валідного JWT-токена, користувач не має доступу до захищених API Routes та сторінок системи, окрім публічних — головної сторінки, каталогу проектів та каталогу фрілансерів. Вихід з облікового запису виконується видаленням JWT-токена з cookie браузера через виклик функції `signOut()` бібліотеки NextAuth.js.

## 9 БЕЗПЕКА ІНФОРМАЦІЙНОЇ СИСТЕМИ

Для забезпечення безпеки на рівні бази даних виконано створення окремих ролей бази даних та видачу їм привілеїв відповідно до функціональних обов'язків кожної групи користувачів.

Ознайомитись зі списком привілеїв ролей на об'єкти БД можна в додатку H.

В інформаційній системі «FreelanceHub» реалізовано 4 групові ролі: Адміністратор (freelance_admin), Замовник (freelance_client), Виконавець (freelance_contractor) та Гість (freelance_guest).

З текстом запитів на створення ролей та привілеїв можна ознайомитись в додатку J.

Для неавторизованого доступу створено службову роль freelance_guest, від імені якої виконується перегляд публічних даних системи — каталогу проектів, профілів виконавців, відгуків та загальної статистики платформи. Ця роль має мінімальний набір привілеїв (SELECT) на таблиці Project, Contractor, Ratings, обмежені стовпці таблиці Client (client_id, first_name, last_name, city) та три представлення: top_contractors, active_projects та project_statistics.

Оскільки система побудована на основі веб-додатку Next.js, аутентифікація та авторизація користувачів реалізовано на двох рівнях: на рівні додатку (через бібліотеку NextAuth.js) та на рівні бази даних (через ролі та привілеї PostgreSQL).

**Аутентифікація на рівні додатку.** При вході в систему користувач вводить логін та пароль на сторінці авторизації. Модуль lib/auth.js (див. лістинг 8.4) виконує пошук облікового запису в таблиці Users та перевірку пароля. Аутентифікація виконується від імені суперкористувача бази даних, оскільки потребує доступу до стовпця password_hash таблиці Users. Паролі зберігаються у базі даних у вигляді хешів, згенерованих алгоритмом bcrypt з бібліотеки bcryptjs. Алгоритм bcrypt автоматично додає сіль (salt) до кожного пароля перед хешуванням, що унеможливлює використання райдужних таблиць (rainbow tables) для зворотного відновлення паролів.

**Розмежування доступу до паролів.** Стовпець password_hash таблиці Users захищено на рівні бази даних за допомогою механізму стовпцевих привілеїв (column-level privileges). Ролі freelance_client та freelance_contractor мають доступ лише до стовпців user_id, login, email, role та created_at таблиці Users. Стовпець password_hash виключено з привілеїв цих ролей, тому спроба виконати SELECT password_hash FROM Users від імені замовника чи виконавця призведе до помилки доступу. Роль freelance_guest не має жодного доступу до таблиці Users. Повний доступ до стовпця password_hash має лише роль freelance_admin та суперкористувач бази даних.

Після успішної перевірки пароля функція authorize() завантажує дані профілю з таблиці Client або Contractor залежно від ролі та формує об'єкт користувача з інформацією про ідентифікатор, роль та ім'я. Цей об'єкт серіалізується у JWT-токен (JSON Web Token), який зберігається у захищеному cookie браузера.

**Авторизація на рівні додатку.** Визначення активної ролі користувача системи відбувається під час послідовних аутентифікації та авторизації. Як зазначено в розділі 8, всередині модуля lib/auth.js колбек jwt зберігає роль користувача у токені, а колбек session передає її в об'єкт сесії. Кожний API Route перевіряє роль через виклик функції getServerSession(authOptions) та порівнює session.user.role з дозволеними ролями для конкретної операції. Наприклад, створення проекту дозволено лише ролі client, а доступ до адмін-панелі — лише ролі admin. Якщо роль не відповідає вимогам, сервер повертає HTTP-відповідь зі статусом 403 (Доступ заборонено).

**Доступ до бази даних через SET ROLE.** Ключовим механізмом розмежування доступу на рівні бази даних є використання команди SET ROLE. Додаток підключається до PostgreSQL від імені суперкористувача через єдиний пул з'єднань (connection pool). Перед виконанням кожного SQL-запиту модуль lib/db.js виконує команду SET ROLE для перемикання на відповідну роль бази даних:

- якщо користувач має роль admin — встановлюється SET ROLE freelance_admin;
- якщо користувач має роль client — встановлюється SET ROLE freelance_client;
- якщо користувач має роль contractor — встановлюється SET ROLE freelance_contractor;
- якщо користувач не авторизований (гість) — встановлюється SET ROLE freelance_guest.

Після виконання запиту з'єднання повертається до початкового стану командою RESET ROLE. Таким чином, PostgreSQL перевіряє привілеї (GRANT) та політики RLS для кожного запиту відповідно до ролі поточного користувача, а не суперкористувача.

**Безпека на рівні рядків (RLS).** Додатково до привілеїв на рівні таблиць та стовпців, в базі даних PostgreSQL налаштовано механізм Row Level Security (RLS) — безпеку на рівні рядків. RLS дозволяє обмежити доступ до окремих рядків таблиці залежно від поточної ролі підключення. Політики RLS створено для трьох таблиць:

- **Project** — перегляд дозволено всім ролям (політика project_select_all), а модифікація (створення, оновлення, видалення) — лише замовнику, який є власником проекту (політика project_modify_own);
- **Deal** — перегляд дозволено лише учасникам угоди (замовнику або виконавцю) та адміністратору (політика deal_select_own);
- **Safe** — перегляд ескроу-рахунку дозволено лише учасникам пов'язаної угоди та адміністратору (політика safe_select_related).

Таким чином, доступ до даних системи контролюється на трьох рівнях: авторизація на рівні додатку (перевірка ролі з JWT-токена), привілеї на рівні бази даних (GRANT на таблиці та стовпці, зокрема захист password_hash) та політики RLS (обмеження доступу до конкретних рядків залежно від ролі).

## ВИСНОВКИ

В результаті аналізу предметної області сформульовано мету створення та визначено основні особливості функціонування ІС «FreelanceHub», список груп користувачів (Гість, Замовник, Виконавець, Адміністратор) та списки їх вимог до створюваної ІС.

Для реалізації курсового проекту обрано трирівневу клієнт-серверну архітектуру та шаблон програмування MVC, СУБД PostgreSQL 18.1, фреймворк Next.js 16.1.6 на мові JavaScript для організації серверної та клієнтської частин додатку, бібліотеку pg (node-postgres) для взаємодії з базою даних через параметризовані SQL-запити, бібліотеку NextAuth.js для організації аутентифікації та авторизації на основі JWT-токенів, а також CSS-фреймворк Tailwind CSS v4 для створення адаптивного користувацького інтерфейсу.

У ході розробки створено базу даних, що містить 10 таблиць, 5 типів ENUM, 1 домен, 3 представлення, 5 функцій та 3 тригери. Серверна частина додатку реалізована у вигляді 13 API Routes, а клієнтська — у вигляді 14 React-сторінок з адаптивним дизайном.

Механізм авторизації реалізовано на двох рівнях: на рівні додатку (NextAuth.js з JWT-токенами та bcrypt-хешуванням паролів) та на рівні бази даних (4 ролі з GRANT-привілеями та 3 таблиці з політиками Row Level Security). Тригери та тригерні функції забезпечують автоматичне оновлення рейтингів виконавців, а параметризовані SQL-запити захищають від SQL-ін'єкцій.

Завдяки рольовому розмежуванню повноважень серед різних груп користувачів забезпечено захист від несанкціонованого доступу до системи.

Створена система надає можливості:

- замовникам створювати проекти з зазначенням спеціалізації, бюджету та дедлайну, укладати угоди з виконавцями, залишати відгуки та подавати скарги;
- виконавцям переглядати доступні проекти, керувати завданнями в рамках угод, оновлювати профіль зі спеціалізацією, резюме та портфоліо;
- адміністраторам переглядати статистику платформи через представлення бази даних, керувати обліковими записами користувачів та переглядати рейтинг найкращих виконавців;
- гостям переглядати каталог проектів та профілі фрілансерів без реєстрації.

Дана система має перспективи для подальшого розвитку: реалізація системи повідомлень між замовниками та виконавцями, додавання оплати через платіжні системи, розширення аналітичної панелі адміністратора та впровадження автоматичного підбору виконавців на основі спеціалізації та рейтингу.

## ДОДАТОК B

### Опис сутностей бази даних інформаційної системи "FreelanceHub"

#### B.1 Сутність «Users» (Користувачі)

Сутність «Users» зберігає облікові дані всіх зареєстрованих користувачів системи та використовується для авторизації та автентифікації.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| user_id | Унікальний ідентифікатор користувача | PRIMARY KEY, SERIAL (автоінкремент) |
| login | Логін користувача для входу в систему | UNIQUE, NOT NULL, VARCHAR(20), CHECK (тільки латинські літери, цифри, крапка та підкреслення) |
| email | Електронна пошта користувача | UNIQUE, NOT NULL, VARCHAR(100) |
| password_hash | Хеш пароля (bcrypt) | NOT NULL, VARCHAR(255) |
| role | Роль користувача в системі | NOT NULL, DEFAULT 'client', тип user_role ('client', 'contractor', 'admin') |
| created_at | Дата та час створення облікового запису | DEFAULT CURRENT_TIMESTAMP |

#### B.2 Сутність «Client» (Замовники)

Сутність «Client» зберігає профільну інформацію про замовників, які публікують проекти та шукають виконавців.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| client_id | Унікальний ідентифікатор замовника | PRIMARY KEY, SERIAL (автоінкремент) |
| user_id | Зовнішній ключ на таблицю Users | UNIQUE, NOT NULL, FOREIGN KEY → Users(user_id) ON DELETE CASCADE |
| first_name | Ім'я замовника | NOT NULL, VARCHAR(50) |
| last_name | Прізвище замовника | NOT NULL, VARCHAR(50) |
| photo | URL фотографії профілю | NULL, VARCHAR(255) |
| city | Місто проживання | NULL, VARCHAR(50) |
| rating | Рейтинг замовника (від 0 до 100) | DEFAULT 0, CHECK (rating BETWEEN 0 AND 100) |
| created_at | Дата та час створення профілю | DEFAULT CURRENT_TIMESTAMP |
| updated_at | Дата та час останнього оновлення профілю | DEFAULT CURRENT_TIMESTAMP |

#### B.3 Сутність «Contractor» (Виконавці)

Сутність «Contractor» зберігає профільну інформацію про фрілансерів, що виконують роботи за проектами.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| contractor_id | Унікальний ідентифікатор виконавця | PRIMARY KEY, SERIAL (автоінкремент) |
| user_id | Зовнішній ключ на таблицю Users | UNIQUE, NOT NULL, FOREIGN KEY → Users(user_id) ON DELETE CASCADE |
| first_name | Ім'я виконавця | NOT NULL, VARCHAR(50) |
| last_name | Прізвище виконавця | NOT NULL, VARCHAR(50) |
| specialization | Спеціалізація (сфера діяльності) | NULL, VARCHAR(100) |
| resume | Резюме виконавця | NULL, TEXT |
| portfolio | Посилання на портфоліо | NULL, TEXT |
| photo | URL фотографії профілю | NULL, VARCHAR(255) |
| city | Місто проживання | NULL, VARCHAR(50) |
| rating | Рейтинг виконавця (від 0 до 100) | DEFAULT 0, CHECK (rating BETWEEN 0 AND 100) |
| created_at | Дата та час створення профілю | DEFAULT CURRENT_TIMESTAMP |
| updated_at | Дата та час останнього оновлення профілю | DEFAULT CURRENT_TIMESTAMP |

#### B.4 Сутність «Project» (Проекти)

Сутність «Project» зберігає інформацію про проекти, що публікуються замовниками для пошуку виконавців.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| project_id | Унікальний ідентифікатор проекту | PRIMARY KEY, SERIAL (автоінкремент) |
| client_id | Зовнішній ключ на таблицю Client (замовник проекту) | NOT NULL, FOREIGN KEY → Client(client_id) ON DELETE CASCADE |
| specialization | Спеціалізація проекту | NOT NULL, тип project_specialization ('Web Development', 'Design', 'Writing', 'Marketing') |
| description | Детальний опис проекту | NOT NULL, TEXT |
| budget | Бюджет проекту (у грн) | NOT NULL, DECIMAL(10,2), CHECK (budget >= 500) |
| deadline | Дата завершення проекту | NOT NULL, DATE |
| created_at | Дата та час створення проекту | DEFAULT CURRENT_TIMESTAMP |
| updated_at | Дата та час останнього оновлення | DEFAULT CURRENT_TIMESTAMP |

#### B.5 Сутність «Deal» (Угоди)

Сутність «Deal» зберігає інформацію про угоди між замовниками та виконавцями в рамках конкретних проектів.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| deal_id | Унікальний ідентифікатор угоди | PRIMARY KEY, SERIAL (автоінкремент) |
| project_id | Зовнішній ключ на таблицю Project | FOREIGN KEY → Project(project_id) ON DELETE SET NULL |
| client_id | Зовнішній ключ на таблицю Client (замовник) | FOREIGN KEY → Client(client_id) ON DELETE SET NULL |
| contractor_id | Зовнішній ключ на таблицю Contractor (виконавець) | FOREIGN KEY → Contractor(contractor_id) ON DELETE SET NULL |
| status | Статус угоди | NOT NULL, DEFAULT 'Pending', тип deal_status ('Pending', 'In Progress', 'Completed', 'Cancelled') |
| amount | Сума угоди | DECIMAL(10,2), CHECK (amount > 0) |
| creation_date | Дата укладання угоди | NOT NULL, DEFAULT CURRENT_DATE |
| completion_date | Дата завершення угоди | NULL, DATE, CHECK (completion_date IS NULL OR creation_date <= completion_date) |
| created_at | Дата та час створення запису | DEFAULT CURRENT_TIMESTAMP |
| updated_at | Дата та час останнього оновлення | DEFAULT CURRENT_TIMESTAMP |

#### B.6 Сутність «ContractorTask» (Завдання виконавців)

Сутність «ContractorTask» зберігає інформацію про конкретні завдання, що виконуються фрілансерами в рамках угод.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| contractortask_id | Унікальний ідентифікатор завдання | PRIMARY KEY, SERIAL (автоінкремент) |
| contractor_id | Зовнішній ключ на таблицю Contractor | NOT NULL, FOREIGN KEY → Contractor(contractor_id) ON DELETE CASCADE |
| deal_id | Зовнішній ключ на таблицю Deal | NOT NULL, FOREIGN KEY → Deal(deal_id) ON DELETE CASCADE |
| task_type | Тип завдання | NULL, VARCHAR(100) |
| input_data | Вхідні дані завдання (JSON) | NOT NULL, DEFAULT '{}', JSONB |
| output_data | Вихідні дані завдання (JSON) | NOT NULL, DEFAULT '{}', JSONB |
| created_at | Дата та час створення завдання | DEFAULT CURRENT_TIMESTAMP |
| — | Обмеження унікальності | UNIQUE (contractor_id, deal_id) — одне завдання на виконавця в рамках однієї угоди |

#### B.7 Сутність «Safe» (Ескроу-рахунки)

Сутність «Safe» зберігає інформацію про фінансові гарантії (ескроу), що забезпечують безпечність угод між замовниками та виконавцями.

| Ім'я атрибута | Призначення атрибута | Обмеження |
|---------------|----------------------|-----------|
| safe_id | Унікальний ідентифікатор ескроу-рахунку | PRIMARY KEY, SERIAL (автоінкремент) |
| deal_id | Зовнішній ключ на таблицю Deal | FOREIGN KEY → Deal(deal_id) ON DELETE CASCADE |
| amount | Сума заморожених коштів | NOT NULL, DECIMAL(10,2), CHECK (amount > 0) |
| status | Статус ескроу-рахунку | NOT NULL, DEFAULT 'Frozen', тип safe_status ('Frozen', 'Released', 'Returned') |
| freeze_date | Дата заморожування коштів | NOT NULL, DEFAULT CURRENT_DATE |
| release_date | Дата вивільнення коштів виконавцю | NULL, DATE |
| return_date | Дата повернення коштів замовнику | NULL, DATE |
| — | Обмеження консистентності статусу та дат | CHECK: якщо статус 'Released' — release_date NOT NULL; якщо 'Returned' — return_date NOT NULL; якщо 'Frozen' — обидві дати NULL |
| created_at | Дата та час створення запису | DEFAULT CURRENT_TIMESTAMP |

#### B.8 Сутність «Ratings» (Рейтинги та відгуки)

Сутність «Ratings» зберігає оцінки та текстові відгуки, що залишають замовники для виконавців після завершення проектів.

| reviewed_id | Зовнішній ключ на таблицю Contractor (об'єкт оцінки) | NOT NULL, FOREIGN KEY → Contractor(contractor_id) ON DELETE CASCADE |

```sql
-- [FUNCTION] Ранжування виконавців за кількістю завершених проектів
-- для конкретної спеціалізації проекту
CREATE OR REPLACE FUNCTION get_ranked_contractors(project_num INT)
RETURNS TABLE(
    contractor_id INT,
    first_name VARCHAR,
    last_name VARCHAR,
    specialization VARCHAR,
    completed_projects BIGINT,
    rank BIGINT
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        c.contractor_id,
        c.first_name,
        c.last_name,
        c.specialization,
        COUNT(d.deal_id) AS completed_projects,
        RANK() OVER (ORDER BY COUNT(d.deal_id) DESC) AS rank
    FROM Contractor c

                    AND d.status = 'Completed'
    WHERE c.specialization::text = (

        FROM Project pr WHERE pr.project_id = project_num
    )
    GROUP BY c.contractor_id, c.first_name, c.last_name, c.specialization

-- [FUNCTION] Розрахунок середньої вартості години роботи над проектом
CREATE OR REPLACE FUNCTION calculate_avg_hourly_cost(
    publication_time TIMESTAMP,
    completion_time TIMESTAMP,
    p_project_id INT
)
RETURNS NUMERIC(10,2) AS $$
DECLARE
    total_hours NUMERIC;
    project_amount NUMERIC;
    avg_hourly_cost NUMERIC(10,2);
BEGIN
    total_hours := EXTRACT(EPOCH FROM (completion_time - publication_time)) / 3600;
    IF total_hours <= 0 THEN
        RAISE EXCEPTION 'Час завершення повинен бути після часу публікації.';
    END IF;

    SELECT amount INTO project_amount
    FROM Deal WHERE Deal.project_id = p_project_id;

    IF project_amount IS NULL THEN
        RAISE EXCEPTION 'Не знайдено контракт з ID проекту: %', p_project_id;
    END IF;

    avg_hourly_cost := project_amount / total_hours;
    RETURN ROUND(avg_hourly_cost, 2);
END;
$$ LANGUAGE plpgsql;
```

```sql
END;
$$ LANGUAGE plpgsql;
```

*Лістинг F.4 — Код функції ранжування виконавців за проектом*

Функція для розрахунку середньої вартості години роботи над проектом (лістинг F.5). Функція приймає три параметри: час публікації проекту, час завершення та ідентифікатор проекту. Обчислює загальну кількість годин роботи, отримує суму угоди з таблиці Deal та повертає середню вартість однієї години. У разі некоректних вхідних даних генерує виключення.

```sql
-- [FUNCTION] Розрахунок середньої вартості години роботи
CREATE OR REPLACE FUNCTION calculate_avg_hourly_cost(
    publication_time TIMESTAMP,
    completion_time TIMESTAMP,
    p_project_id INT
)
RETURNS NUMERIC(10,2) AS $$
DECLARE
    total_hours NUMERIC;
    project_amount NUMERIC;
    avg_hourly_cost NUMERIC(10,2);
BEGIN
    total_hours := EXTRACT(EPOCH FROM
        (completion_time - publication_time)) / 3600;
    IF total_hours <= 0 THEN
        RAISE EXCEPTION
            'Час завершення повинен бути після часу публікації.';
    END IF;

    SELECT amount INTO project_amount
    FROM Deal WHERE Deal.project_id = p_project_id;

    IF project_amount IS NULL THEN
        RAISE EXCEPTION
            'Не знайдено контракт з ID проекту: %', p_project_id;
    END IF;

    avg_hourly_cost := project_amount / total_hours;
    RETURN ROUND(avg_hourly_cost, 2);
END;
$$ LANGUAGE plpgsql;
```

Створення політик Row Level Security (RLS) для обмеження доступу до окремих рядків:

```sql
-- RLS для таблиці Project
ALTER TABLE Project ENABLE ROW LEVEL SECURITY;

CREATE POLICY project_select_all ON Project
    FOR SELECT USING (true);

CREATE POLICY project_modify_own ON Project
    FOR ALL USING (
        client_id IN (
            SELECT c.client_id FROM Client c
            JOIN Users u ON c.user_id = u.user_id
            WHERE u.login = current_setting(
                'app.current_login', true)
        )
    );

-- RLS для таблиці Deal
ALTER TABLE Deal ENABLE ROW LEVEL SECURITY;

CREATE POLICY deal_select_own ON Deal
    FOR SELECT USING (
        client_id IN (
            SELECT c.client_id FROM Client c
            JOIN Users u ON c.user_id = u.user_id
            WHERE u.login = current_setting(
                'app.current_login', true))
        OR contractor_id IN (
            SELECT co.contractor_id FROM Contractor co
            JOIN Users u ON co.user_id = u.user_id
            WHERE u.login = current_setting(
                'app.current_login', true))
        OR current_user = 'freelance_admin'
    );

-- RLS для Safe
ALTER TABLE Safe ENABLE ROW LEVEL SECURITY;

CREATE POLICY safe_select_related ON Safe
    FOR SELECT USING (
        deal_id IN (
            SELECT d.deal_id FROM Deal d
            WHERE d.client_id IN (
                SELECT c.client_id FROM Client c
                JOIN Users u ON c.user_id = u.user_id
                WHERE u.login = current_setting(
                    'app.current_login', true))
            OR d.contractor_id IN (
                SELECT co.contractor_id FROM Contractor co
                JOIN Users u ON co.user_id = u.user_id
                WHERE u.login = current_setting(
                    'app.current_login', true))
        )
        OR current_user = 'freelance_admin'
    );
```
